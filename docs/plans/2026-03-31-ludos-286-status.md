# LUDOS-286 Restart Notes

Date: 2026-04-01

This file is a restart note for continuing `LUDOS-286` after a break or a crashed session.

## Goal

Reduce the AWS permissions used by the CI/CD deployment lane while keeping:
- local developer deploys possible with full permissions via `--full-developer-permissions`
- local default deploys aligned with the restricted CI lane
- GitHub Actions and local CI simulation using the same reduced-permission model

## Current Design

There are now two deploy lanes:
- restricted CI lane: default for local deploy scripts and intended for GitHub Actions
- full developer lane: enabled with `--full-developer-permissions`

The restricted CI lane uses:
- restricted deploy role
- restricted CloudFormation execution role
- permissions boundary that caps what restricted CI roles can ever do

The utility account also has a separate restricted image-read role for reading the latest image from ECR during the untuva deploy flow.

## Key Behavioral Rules Implemented

- All deploy entry scripts support `--full-developer-permissions`.
- Local deploys default to the restricted CI lane.
- If a restricted deploy fails, deploy scripts print a reminder:
  - `Restricted deploy failed. Retry with --full-developer-permissions if you need broader permissions.`
- Full-permission deploys export concrete AWS credentials from the selected SSO profile so `cdk deploy` works reliably.
- Utility deploys no longer require `IMAGE_TAG=dummy`.

## What Was Implemented

### Shared deploy-mode support

Implemented shared support for:
- parsing `--full-developer-permissions`
- selecting restricted vs full credential path
- assuming restricted roles locally
- exporting SSO-backed credentials for host-side CDK use

### Restricted CI role chain

Implemented restricted CI roles in target accounts:
- `ludos-restricted-ci-deploy-role`
- `ludos-restricted-ci-cfn-exec-role`

Implemented restricted CI roles in the utility account:
- `ludos-restricted-ci-image-read-role`
- `ludos-restricted-ci-deploy-role`
- `ludos-restricted-ci-cfn-exec-role`

### Permissions boundary

Added permissions boundary:
- `ludos-restricted-ci-permissions-boundary`

The boundary is attached to the restricted CI roles so the restricted lane cannot exceed that ceiling even if someone later broadens an inline policy.

### Trust policy hardening

Restricted CI roles are trusted only from the AWS Identity Center `AdministratorAccess` role pattern in the same account, using both:
- IAM role ARN pattern
- STS assumed-role ARN pattern

Helper extracted in `infra/lib/githubActionsStack.ts`:
- `createRestrictedCiRoleAssumerPrincipal(accountId: string)`

### Utility role split

The utility account originally used the image-read role where a deploy role was needed. This was fixed by splitting:
- image-read role for ECR lookup only
- deploy role for actual utility stack deployment

Important CloudFormation detail:
- the existing utility image-read role had to keep its old construct logical ID lineage
- `ecrStack.ts` was adjusted so the old role stayed under the old logical ID and the new deploy role got a new logical ID
- this avoided a `role already exists in stack` failure

### Deploy failure hint propagation

Deploy entry scripts now use `set -o errtrace` so the restricted-mode reminder trap also fires through functions and sourced scripts like `infra/cdk.sh`.

- removed `PowerUserAccess` from the restricted CI CloudFormation execution role
- replaced it with explicit allowed actions plus a permissions boundary

## Known Debugging Lessons From This Session

- If a role trust policy allows only the IAM role ARN form, local SSO users can still fail because the actual caller is an STS assumed-role ARN.
- If restricted utility deploy fails on bootstrap/SSM checks, make sure the script is using the utility restricted deploy role, not the image-read role.
- If `cdk deploy` says credentials are missing even though the AWS CLI check passed, exporting concrete AWS credentials from the chosen SSO profile is more reliable than relying on profile resolution inside Node/CDK.
- If CloudFormation says a role already exists in the stack, check whether a construct rename changed the logical ID for an already-existing physical role.
- If the deploy hint does not print on failures inside sourced scripts/functions, `set -o errtrace` is required in the deploy entry scripts.
